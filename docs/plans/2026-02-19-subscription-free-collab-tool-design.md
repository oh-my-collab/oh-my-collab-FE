# 구독 없는 협업 툴 아이디어 고도화 (캡스톤 팀 대상)

## 결정 사항 (2026-02-19)
- 접근 방식: **옵션 A (Vercel + Supabase/Neon 무료 티어)** 채택
- 이유: 배포 난이도, 기능 확장성, 무료 유지 가능성의 균형이 가장 좋음

## 1) 목표와 문제 정의
- 목표: 월 구독료 없이 팀별로 자체 호스팅해서 쓰는 협업 툴 제공
- 핵심 문제: 기존 협업툴은 편하지만 비용이 누적되고, 학생 팀/소규모 팀은 비용 부담이 큼
- 해결 방식: Vercel 무료 플랜 + 오픈소스 템플릿을 기반으로 팀이 직접 1회 배포

## 2) 타겟 사용자
- 1차 타겟: 캡스톤 프로젝트 팀, 동아리 프로젝트 팀, 스터디 팀
- 페르소나: "돈은 아끼고 싶지만 초기 설정 30~90분은 감수 가능한 사용자"
- 비타겟: 설정/운영을 완전히 위임하고 싶은 사용자, 강한 엔터프라이즈 보안 요구 조직

## 3) 제품 포지셔닝
- 한 줄 포지셔닝: "설정은 조금 번거롭지만, 이후 비용은 거의 0원에 가까운 팀 협업 베이스캠프"
- 차별점:
  - 구독형 SaaS가 아닌 팀 소유형 배포 모델
  - 팀별 독립 인스턴스(데이터 분리 명확)
  - 캡스톤에 맞춘 기능(기여도/목표 추적/문서화)

## 4) 접근 방식 옵션 (2~3안)
### 옵션 A: Vercel + Supabase(또는 Neon) 무료 티어 (권장)
- 구성: Next.js(프론트 + API Route) + Managed Postgres + Auth
- 장점: 배포/운영 난이도와 기능 확장성의 균형이 가장 좋음
- 단점: 외부 서비스 계정 연결 필요, 무료 티어 한도 관리 필요

### 옵션 B: Vercel + 파일 기반 저장(JSON/Markdown)
- 구성: 정적/반정적 앱 + Git 저장소를 데이터 저장소로 사용
- 장점: 구조가 단순하고 시작이 빠름
- 단점: 동시성, 권한, 통계 정확도, 확장성 한계가 큼

### 옵션 C: 팀별 VPS 셀프호스팅 (Docker)
- 구성: 풀스택 컨테이너 배포
- 장점: 자유도 최고
- 단점: 비용/운영 난이도 증가, "무료" 메시지 약화

### 권장안
- 옵션 A 권장. 이유: 무료 유지 가능성과 "협업툴다운 기능" 구현 난이도 사이 균형이 가장 좋음.

## 5) 핵심 기능 설계
### 기능 1. 문서화
- 팀 위키(회의록, 결정사항, 트러블슈팅)
- 문서 템플릿(회의록/주간보고/회고)
- 버전 이력 및 변경자 표시

### 기능 2. 통계 대시보드
- 주간 완료 태스크 수
- 목표 달성률
- 개인/팀 활동량 추이
- 마감 임박 항목 수

### 기능 3. 할 일 정리
- 칸반 보드(To do / In progress / Done)
- 담당자, 우선순위, 마감일
- 반복 태스크/체크리스트

### 기능 4. 목표 추적 및 관리
- 목표(Goal) > 핵심결과(Key Result) 구조
- 주차별 진척률 입력
- 리스크/블로커 기록

### 기능 5. 팀 기여도 측정
- 입력 신호:
  - 태스크 완료 수(난이도 가중치 포함)
  - 문서 업데이트 기여
  - 목표 진행 업데이트 빈도
  - 협업 행동(코멘트/리뷰/블로커 해결)
- 출력:
  - 개인 기여도 점수(정규화)
  - 팀 내 상대 기여 분포
  - 주간 변화량

## 6) 기여도 점수 모델 (초안)
- 점수 예시:
  - `contribution_score = 0.40*task_score + 0.20*docs_score + 0.25*goal_score + 0.15*collab_score`
- 보정 규칙:
  - 단순 수량 과대평가 방지를 위해 난이도/영향도 가중치 적용
  - 팀 리더가 분기별로 가중치 조정 가능
  - 점수는 "평가"가 아니라 "가시화" 용도임을 UI에서 명시

## 7) 정보 구조 (IA)
- Workspace
- Projects
- Docs
- Tasks
- Goals
- Insights (통계/기여도)
- Settings (배포, 멤버, 가중치)

## 8) MVP 범위 (캡스톤 1학기 기준)
- 포함:
  - 문서화 기본
  - 칸반 태스크
  - 목표/KR 관리
  - 통계 대시보드 기본
  - 기여도 점수 1차 모델
- 제외:
  - 실시간 협업 편집
  - 고급 권한 체계(RBAC 세분화)
  - 외부 툴 심화 연동(Slack/Discord 양방향 자동화)

## 9) 온보딩/배포 경험
- 팀 리더 기준 15분 온보딩 목표:
  1. 템플릿 레포 포크
  2. Vercel Import
  3. DB 연결
  4. 초기 워크스페이스 생성
  5. 팀원 초대
- 필수 산출물:
  - "원클릭 배포 가이드" 문서
  - 환경변수 체크리스트
  - 샘플 데이터 시드 스크립트

## 10) 리스크와 대응
- 리스크: 설정이 번거로워 첫 이탈 발생
  - 대응: 설치 마법사형 온보딩 + 체크리스트 자동 검증
- 리스크: 기여도 점수 공정성 논란
  - 대응: 점수 공식 공개, 팀별 가중치 커스터마이즈, 원본 데이터 함께 노출
- 리스크: 무료 티어 한도 초과
  - 대응: 데이터 보관 주기/압축, 사용량 경고, 백업 가이드 제공

## 11) 성공 지표 (MVP)
- 설치 완료율: 70%+
- 첫 주 활성 팀 비율: 60%+
- 주간 재방문율: 50%+
- 문서/태스크/목표 중 2개 이상 기능 동시 사용 팀 비율: 65%+

## 12) 실행 우선순위 (추천)
1. 배포/온보딩 자동화(가장 큰 이탈 포인트 제거)
2. 태스크 + 목표 + 문서 기본 루프 완성
3. 통계/기여도 시각화 추가
4. 캡스톤 제출물 템플릿(주간보고/최종보고) 연동

## 13) 다음 결정 필요 항목
- 기여도 지표에 코드 활동(Git 연동)을 MVP에 포함할지 여부
- 팀당 사용자 수 상한과 데이터 보존 기간 기본값
- 학기 종료 후 아카이브 정책(읽기 전용 전환 vs 삭제)
